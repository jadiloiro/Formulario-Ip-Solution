# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

IP Solution's onboarding tool: a multi-step survey form (client queues, agents, hours, BOT config) plus a visual chatbot-flow editor (Drawflow), backed by a small NestJS API. Code, comments and UI text are in Portuguese (pt-BR) — match that when editing.

## Commands

`npm test` (unit, Jest, mocked repositories), `npm run test:e2e` (supertest against a real Postgres — needs one reachable, see below), and `npm run lint` (ESLint flat config) all exist in `backend/package.json`.

Server runs at `http://localhost:3000`. The frontend is served directly from `backend/public/` by the same Nest process (`ServeStaticModule`), so there is no separate frontend dev server and no build step for the frontend — HTML/CSS/JS in `public/` are edited and served as-is.

Needs a running Postgres — either `docker compose up -d postgres` (repo root) or your own instance, configured via `backend/.env` (copy from `backend/.env.example`). Migrations run automatically on boot (`migrationsRun: true`); to manage them manually: `npm run migration:generate -- src/migrations/Name`, `npm run migration:run`, `npm run migration:revert`.

## Architecture

### Backend (`backend/src/`)
NestJS app, four feature modules:
- `app.module.ts` wires `TypeOrmModule.forRootAsync` (options from `src/config/database.config.ts`'s `buildDataSourceOptions()`, reading `DB_HOST`/`DB_PORT`/`DB_USERNAME`/`DB_PASSWORD`/`DB_DATABASE` env vars) and `ServeStaticModule.forRoot({ rootPath: 'public' })`. Postgres only — `synchronize` is `false`; schema changes go through TypeORM migrations in `src/migrations/`, applied automatically on boot (`migrationsRun: true`). `src/data-source.ts` is the standalone `DataSource` the TypeORM CLI uses for `npm run migration:generate/run/revert` (reuses the same `buildDataSourceOptions()`, loads `.env` itself since it runs outside Nest's DI).
- `main.ts`: `cookie-parser` middleware (reads the httpOnly session cookie), CORS enabled with `credentials: true`, global `ValidationPipe({ whitelist: true, transform: true })`, and `app.setGlobalPrefix('api', { exclude: [''] })` — so `/` still serves the static frontend but all controller routes live under `/api/*`.
- `users/`: the `User` entity — accounts are provisioned only by a `super_admin` (the Implantação team), never self-registered. Columns: `login` (unique), `passwordHash` (argon2id), `mustChangePassword` (forces a password change on first login), `role` (`'super_admin' | 'cliente'`), `moduleWhatsapp`/`moduleTelefonia` (per-client module entitlements set at creation), `clientName`. `auth/users.controller.ts` (super_admin only) also has `GET /api/users` (lists all `'cliente'` accounts, never super_admins, never `passwordHash` — feeds `clientes.html`), `PATCH /api/users/:id` (edit `clientName`/`login`/modules; optional `novaSenha` re-forces `mustChangePassword`), and `DELETE /api/users/:id` — both go through `UsersService`'s private `findClientOrThrow()`, so a super_admin id can never be edited/deleted through this endpoint even if guessed.
- `auth/`: session-based auth — `POST /api/auth/login` validates against `UsersService` and opens a row in the `Session` entity (`sessions` table), returning an **opaque random token in an httpOnly/Secure/SameSite=Strict cookie** (not a JWT — see `auth.service.ts` for why: revocability). `SessionGuard` reads that cookie on every request, validates it against `sessions`, and populates `request.user`; it also blocks any route (except ones marked `@SkipPasswordCheck()`) while `mustChangePassword` is true. `RolesGuard` + `@Roles('super_admin')` gate super_admin-only routes (e.g. `POST /api/users`, the client-provisioning endpoint). `AuthService.assertModuleAccess()` is the RBAC check for onboarding modules, called from `submissions/` before accepting a write for a given `onboardingType`.
- `submissions/` is the client-data module: one entity (`Submission`) with `formData` and `flowData` stored as `simple-json` blobs (no normalized schema — the form's shape lives in the frontend, not in migrations), plus `userId` (owner — the real multi-tenancy boundary) and `onboardingType` (`'whatsapp' | 'telefonia'`, checked against the owner's module entitlements). `status` is `'rascunho' | 'enviado'`. Every controller method requires a valid session; non-`super_admin` lookups are always scoped by `userId` in the query itself (`findOneForUser`, `submissions.service.ts`) and return `404` (not `403`) on cross-tenant access, to avoid confirming another client's submission exists.
- `GET /api/submissions/current` is the key endpoint the frontend treats as its "session": it returns the most recent draft (`rascunho`) belonging to the authenticated user, or creates an empty one owned by them.
- `PUT /api/submissions/:id/flow` saves only the Drawflow graph (`flowData`), separate from the general `PATCH /api/submissions/:id` used for form fields.
- `POST /api/submissions/:id/submit` flips status to `enviado` — the only state transition modeled.
- `GET /api/submissions` (all submissions — "painel da equipe") is restricted to `super_admin`.
- `submissions/attachments.{controller,service}.ts` + `entities/attachment.entity.ts`: files a client attaches to a step (`attachments` table, FK `submissionId` `ON DELETE CASCADE`). Binaries live on disk under `backend/uploads/<submissionId>/<uuid><ext>` (never served by `ServeStaticModule` — only via the authenticated `GET /api/submissions/:id/attachments/:attachmentId/download`, which always forces `Content-Disposition: attachment`). Upload is `multipart/form-data` via `FileInterceptor('file', { storage: memoryStorage() })` (20MB limit); ownership is checked the same way as the rest of `submissions/` (`SubmissionsService.findOneForUser`, reused by `AttachmentsService`) before touching disk or DB. `SubmissionsService.remove()` also deletes the submission's upload directory.

### Frontend (`backend/public/`)
Plain HTML/CSS/JS, no framework, no bundler. Libraries (Drawflow, Font Awesome) are loaded from CDN in the HTML `<head>`, not via npm.
- `login.html` + `login.js`: login form + the forced first-login password-change form. `login.js`'s `destinationFor(user)` sends `super_admin` to `clientes.html` and `cliente` to `index.html`.
- `clientes.html` + `clientes.css` + `clientes.js`: the single `super_admin`-only "Área ADM" page (there is no separate `admin.html` anymore — it was merged in). Two tabs toggled client-side (no navigation): **"Clientes"**, the Implantação team's dashboard — one card per client (`GET /api/users` + `GET /api/submissions`, most-recent submission per `userId`), each showing a 7-dot step tracker (mirrors the onboarding sidebar's icons/order, driven by `formData.completedSteps`), status badge, module chips, an expandable list of that submission's attachments (`GET /api/submissions/:id/attachments`) with download links, edit/delete icon buttons, and a link straight to `resumo.html?id=...`; and **"+ Cadastrar cliente"**, the same form used for both creating (`POST /api/users`) and editing (`PATCH /api/users/:id`, prefilled by the card's edit button — password field becomes optional "nova senha") a client. Deleting (`DELETE /api/users/:id`) goes through a confirmation modal (`.cli-modal-overlay`). `resumo.html`'s "voltar" link is role-aware: `super_admin` goes back to `clientes.html`, everyone else to `index.html`.
- `index.html` + `script.js` + `style.css`: the 7-step onboarding form (`currentStep`/`totalSteps` in `script.js`). Step definitions, validation (`validateStep`), draft collection (`collectDraft`) and restoration (`restoreDraft`) all live in this one large `script.js` file.
- `Flowchart.html` + `Flowchart.js` + `Flowchart.css`: a separate page hosting a Drawflow instance that builds the chatbot's action-block flow from the queues/agents captured in the main form. Not currently linked from `index.html` — reachable only by navigating to it directly. References must match the files' exact casing (capital `Flowchart.*`) since deploy targets are Linux (case-sensitive filesystem), unlike local Windows dev.
- **Login is mandatory — there is no offline-without-backend mode anymore.** `shared.js` (loaded by every protected page) exposes `requireAuth()`: it calls `GET /api/auth/me` (the session cookie is httpOnly, so it can't be checked synchronously from JS) and redirects to `login.html` if there's no valid session or the account still has `mustChangePassword` set. `index.html`/`Flowchart.html` set `document.documentElement.style.visibility = 'hidden'` in an inline `<head>` script and only reveal the page once `requireAuth()` resolves, to avoid flashing protected content. `apiCtx`/`initApiSync` (`script.js`, top of file) then fetch/create the user's own submission via `/api/submissions/current` (`credentials: 'include'` on every `/api/*` call, so the cookie is sent) and mirror changes to it (`pushDraftToApi`, `PUT .../flow`, `POST .../submit`); `localStorage` (`STORAGE` keys: `currentStep`, `theme`, `ipsolution_form_draft`, `ipsolution_shared_flow_data`; flow graph under `ipsolution_flow_v2`) still absorbs transient network failures during a session, but is no longer a substitute for being logged in. `initOnboardingGate(user)` also disables the WhatsApp option client-side when `!user.moduleWhatsapp` — cosmetic only, the real enforcement is `AuthService.assertModuleAccess()` on the backend.
- Finishing the 7th step no longer generates a PDF client-side. It finalizes the submission (`POST /api/submissions/:id/submit`) and redirects to `resumo.html?id=<submissionId>` — a standalone page (own `resumo.css`/`resumo.js`, not a dependency of `script.js`) that fetches `GET /api/submissions/:id` and renders every section (filas, agentes, horários, configurações, números, agenda, resumo em texto do fluxo do BOT) as plain HTML/CSS, with `break-inside: avoid` on every card/table so it never splits mid-block. `backend/scripts/gerar-pdf-resumo.js` (`npm run pdf:resumo`, Playwright devDependency) drives a real login + minimal form fill and calls `page.pdf()` on that same rendered page (`emulateMedia('screen')` so no `@media print` rule can alter the layout) — the PDF is a faithful print of what the browser actually shows, not a hand-drawn document. Per-step file attachments (the "Anexos" panel in the step toolbar) upload for real to `POST /api/submissions/:id/attachments` (see backend `submissions/attachments.*` below) — they used to only live in an in-memory JS object and vanished on refresh.

## Deployment

Production target is a Debian VM running everything **natively** (no Docker) — Postgres installed via `apt`, the API run directly with Node (`backend/deploy/ipsolution.service` is a systemd unit template: `WorkingDirectory`, `EnvironmentFile` and `User` are placeholders, adjust to the real install path). `DB_HOST=localhost` in `backend/.env` since Postgres and the app share the same machine.

Steps on the VM: `apt install postgresql`, create a role/db, `npm ci && npm run build` in `backend/`, `npm run migration:run` once, then start via the systemd unit (`systemctl enable --now ipsolution`) or `pm2`.

`docker-compose.yml` + `backend/Dockerfile` (repo root / `backend/`) are kept as an alternative — e.g. for spinning up just Postgres locally during development (`docker compose up -d postgres`) without installing it on your machine. They are not the deploy path currently in use.

## Repo hygiene gotchas

- `.gitignore` (repo root) excludes `backend/node_modules/`, `backend/dist/`, `backend/coverage/`, `backend/uploads/` (client-attachment binaries — see `submissions/attachments.*`), and `.env` files — but they were committed to git history before it existed, so a fresh `git clone` won't have them; `npm install` and copying `.env.example` are still required after cloning.
